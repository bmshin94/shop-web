# 코드 주석 작성 규칙 (Code Commenting Guidelines)

에이전트는 코드를 작성하거나 수정할 때 반드시 아래의 주석 규칙을 준수해야 한다.

### 1. 핵심 원칙: '무엇(What)'이 아닌 '왜(Why)'를 설명할 것
- 코드를 그대로 읽어주는 의미 없는 주석은 철저히 배제한다.
- **Bad:** `// 변수 i를 1 증가시킨다.`
- **Good:** `// 결제 게이트웨이(PG) 응답 지연 시 최대 3회까지 재시도하기 위한 카운터`

### 2. 언어별 표준 문서화 블록(DocBlock) 강제
함수, 클래스, 메서드를 생성할 때는 반드시 각 언어의 표준 규격을 따른다.
- **PHP (Laravel):** 모든 컨트롤러와 서비스 클래스의 메서드 상단에 `PHPDoc`을 작성한다. `@param`, `@return`, `@throws`를 명시한다.
- **JavaScript (jQuery/AJAX):** 복잡한 로직이나 재사용되는 함수 상단에 `JSDoc`을 작성한다.

### 3. 프론트엔드 (HTML/CSS) 주석 영역 구분
- **HTML:** 의미 있는 구역(Section)이 시작되고 끝날 때 명확한 구분자를 넣는다.
  - 예: ``
- **CSS:** Bootstrap 5의 기본 스타일을 오버라이딩(덮어쓰기)하는 커스텀 CSS 위에는 반드시 그 이유를 명시한다.
  - 예: `/* Bootstrap 기본 btn-primary 색상을 브랜드 컬러로 덮어쓰기 */`

### 4. 비즈니스 로직의 단계별 주석 (Inline Comments)
쇼핑몰의 장바구니, 결제, 재고 차감 등 중요한 비즈니스 로직 내부에는 흐름을 파악하기 쉽게 번호를 매겨 주석을 작성한다.
- 예시:
  `// 1. 선택한 상품의 현재 재고량 확인`
  `// 2. 장바구니 세션에 상품 정보 추가`
  `// 3. 총 결제 예상 금액(할인율 반영) 재계산`

### 5. 특수 태그(Annotation Tags)의 통일
유지보수를 위해 미완성된 코드나 주의가 필요한 곳에는 영문 대문자 태그를 사용한다.
- `// TODO:` : 향후 구현해야 할 기능 (예: `// TODO: 무통장 입금 확인 로직 추가 예정`)
- `// FIXME:` : 버그가 있거나 수정이 필요한 코드
- `// NOTE:` : 다른 개발자나 AI가 코드를 읽을 때 참고해야 할 중요한 특이사항
- `// WARNING:` : 절대 수정하면 안 되는 민감한 로직 (예: 결제 금액 검증 로직)

### 6. 언어 통일
- 모든 주석의 설명은 **한국어**로 작성하여 가독성을 높인다. 단, 표준 태그(`@param`, `TODO` 등)와 기술 용어는 영문을 사용한다.
